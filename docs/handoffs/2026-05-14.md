## Decisions made and why

1. **`embed()` returns `(number[] | null)[]` aligned to inputs** (PR #36). Considered three shapes: A) null-padded aligned array, B) throw on empty inputs, C) return an `indices[]` alongside compact vectors. Picked A because the caller just iterates `inputs.forEach((s, i) => vec = result.vectors[i])` and skips on null — zero external book-keeping. B was too brittle for a hot path called every ingestion tick; C requires two arrays the caller has to keep in sync. The empty-string branch still skips the API call (OpenAI 400s on `""`).

2. **sender_map case-folding done on read AND write AND backfill** (PR #37). One-layer fix is dangerous: lowercase-on-read alone would miss mixed-case stored rows from before the fix; lowercase-on-write alone would miss mixed-case stored rows from BEFORE the writer change; backfill alone would miss new ones. Did all three so the routing path is monotone-correct regardless of when a row was registered.

3. **Modify migration 0004 in place rather than add migration 0005** (PR #39). Considered both. In-place edit is the right call because (a) `apply-d1-migrations.sh` keys by filename — production's already-applied 0004 row prevents re-execution; (b) the only meaningful change targets the pre-applied path (fresh D1 / restored backup); (c) a new 0005 that only dedupes would be a no-op on prod and add a confusing two-step provenance story. Sub-agent reviewer confirmed safe.

4. **ADR-006 Whisper choice accepted provisionally** (PR #34). Picked `large-v3` via `faster-whisper` INT8. Could not actually benchmark — no Cantonese audio sample yet. Decision is reaffirm-or-revise at Phase 6 PR 1 against a real sample. Trade-off documented in the ADR: provisional commitment over blocking Phase 2 PR 1 on an audio benchmark.

5. **Normalisation is a pure transformation, not a DB writer** (PR #35). `normalizeItem` returns `{ item, llm: LlmCallResult }` so the caller threads `run_id`/`stage` into `recordCost`. Keeps the module DB-free (testable without Postgres) and lets the orchestrator (Phase 3 PR 4) correlate cost with a specific run.
