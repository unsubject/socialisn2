## Decisions made and why

1. **Phase 0-2 audit via three parallel general-purpose subagents.** Each got one phase, the rubric (must-fix / should-fix / nice-to-fix / healthy), and a list of files to read. Output: a single markdown table per agent. Worked well — the synthesis was straightforward and the agents kept to format. The cost vs running it sequentially in the main session was small but real; payoff was speed.

2. **Split the fixes into FOUR sequential PRs off `main`, not one mega-PR.** PR 44 = Phase 1 correctness, PR 45 = Phase 2 hardening, PR 46 = source seeds (migration 010), PR 47 = docs + nice-to-fix bundle. Reason: each is a coherent review surface, and the per-memory `github_squash_stacked_pr_conflict` says stacked PRs break on squash. Sequential off-main avoids that entirely.

3. **For migration 010 source seeds: add what I could URL-verify, defer the rest as Build tasks.** SPEC §6.3 / §6.5 list many outlets but a chunk of them (The Times UK, The Australian, Crikey, NEJM open abstracts, IEA, BloombergNEF, China Books Review, Globe and Mail politics/business sections, etc.) didn't have an obviously-confirmable public RSS URL at seed time. Putting unverified URLs in a seed migration creates silent 404 noise; the umbrella Build task (`aEkwZ1ZuS2lGaFQyalRUeQ`) tracks them for follow-up per-outlet research.

4. **Smoke fixture: install Postgres + pgvector via Homebrew, point `LITELLM_BASE_URL` at Google's OpenAI-compat endpoint for this one run, commit LiteLLM-in-docker-compose separately.** No Docker on this Mac (no docker / orbstack / colima / podman). PG via brew + a direct Gemini OAI-compat URL kept the smoke unblocked. Cost was $0.000861 / 5 articles — well within "negligible". The LiteLLM-in-compose change is the *correct* architecture and landed as PR 48; smoke just used a temp shortcut.

5. **`postgresql@16` → `postgresql@17` for the local smoke PG.** Homebrew's `pgvector` bottle ships extension files for `postgresql@17` + `postgresql@18` only, not `@16`. Switching to `@17` saved ~5 minutes of building pgvector from source against `@16`. CI still runs against `pg16` via the `pgvector/pgvector:pg16` Docker image; only the local smoke shifted.

6. **Don't commit `scripts/smoke-phase2.ts` to the repo.** It's a throwaway by design — the user originally framed it that way ("Manual smoke fixture"). The whole point was a one-time end-to-end verification, not maintained CI. The script is gone with the temp clone. If we want a repeatable smoke later, that's a separate "scripts/" PR.

## Tried and abandoned

- **`tmpfs` / shell-sourced `.env` for the smoke run.** Zsh interprets unquoted asterisks in cron values (`0 5 * * *`) as globs and treated them as commands. Fell back to letting `dotenv` load `.env` from cwd at process start — clean.

- **Trusting postgres-js tagged-template return types.** The audit-fix path (PR 44) had already flagged that drizzle's raw `db.execute<T>` returns timestamptz as ISO strings in this stack. The smoke fixture stumbled on the same issue *again* — `client\`SELECT … published_at … FROM raw_items\`` returned a string for `published_at`, which then blew up `assignCluster` because `input.publishedAt.toISOString()` is not a function. Memory [[drizzle-pg-execute-timestamp-string]] already captures this; the fix is `new Date(r.published_at)` at every call site. Worth re-flagging because the Phase 3 PR 4 orchestrator will hit it again.

- **Initial `postgresql@16` install for the smoke PG.** Wasted ~3 min before realizing pgvector's brew bottle doesn't ship `@16` extension files. Should have checked `find /opt/homebrew -name vector.control` first.
