## Decisions made and why

1. **Blocker 2 dissolved as false alarm without writing code.** On
   inspection: `src/lib/two_brain_client.ts` (13KB, retry + SSE + graceful
   fallback) already exists, wired into `src/scoring/archive.ts` Stage 5,
   with `tests/lib/two_brain_client.test.ts` coverage. The 2nd-brain side
   also has `archive_search`, `archive_search_text`, `record_pick`, and
   `record_episode_link` deployed in `mcp-worker/` with migration 014's
   `editorial_pick` table. SPEC §13 confirms backfill needs nothing new
   from 2nd-brain. Pre-session assessment was wrong — verified before
   shipping any cross-repo work.

2. **ADR-011 → ADR-012 supersede before writing any backfill code.**
   The first attempt at the RSS-history decision (ADR-011) committed to
   "skip RSS half, keep GDELT half" on the reasoning that GDELT has a
   queryable backlog. Advisor caught it: `src/ingestion/gdelt.ts` exposes
   only per-cluster enrichment primitives (`fetchGkg`,
   `lookupOrFetchCoverage`), not a discovery firehose. Turning GDELT
   into a 30-day discovery source needs a topic-seed query layer that
   doesn't exist and isn't a one-PR design. ADR-012 supersedes by
   restoring the user's original singular framing ("skip historical,
   observe forward") and pulling the GDELT half out too. ADR-011 kept
   in-repo with a Superseded banner — the reasoning gap is itself
   interesting history.

3. **Two PRs instead of an amended ADR-011.** A reviewer reading
   "ADR-011 merged, then ADR-012 supersedes 10 min later" sees the system
   catch its own error; an inline patch would read as "we wrote code that
   didn't match the ADR and patched the ADR to fit." Bias-free audit
   trail matters here because the user explicitly asked for unbiased
   reviews this session.

4. **`probeArchiveSearch` uses `attemptMcpCall` directly (single
   attempt), bypassing the `callMcpTool` retry loop.** The graceful
   wrappers (`archiveSearch`, `recordPick`) swallow all errors and
   return empty results — that's the contract Stage 5 wants but the
   wrong contract for a backfill reachability probe (we need a quick
   yes/no, not "did the retry budget exhaust"). Three-state return
   (`available` / `unreachable` / `not_configured`) lets the
   `backfill_run` row record WHICH failure mode happened.

5. **`not_configured` is intentionally NOT an error; `unreachable` is.**
   `not_configured` means env vars unset — an intentional deploy state
   (e.g., dev environment), not a failure. `unreachable` means env
   configured but the call failed — that's a real degradation. Both go
   into `brain_corpus_status` column, but only `unreachable` contributes
   to `backfill_run.error`. Trade-off flagged for Phase 5 PR 2: deploy
   script should explicitly assert `brain_corpus_status === 'available'`
   so a misconfigured prod deploy doesn't silently pass as "completed"
   with `not_configured`.

6. **`runBackfill` always returns `status='completed'` unless the DB
   UPDATE itself fails.** Per-component degradations (YouTube fetch
   throws, brain probe unreachable) get recorded in the row, not
   surfaced as run failure. Mirrors the existing `runScoring` pattern
   in `src/orchestrator/run.ts` — the convention is "row records the
   degraded state; the run finishes regardless."

7. **Migration 013 columns are all NULLable with closed-set CHECK
   constraints reserving room for non-`'skipped'` values.** A future
   ADR can re-enable RSS-historical (via Wayback / NewsAPI) or
   GDELT-discovery (via topic-seed queries / BigQuery) without a
   schema migration — just add a new value to the CHECK list. Forward-
   compat at zero v1 cost.
