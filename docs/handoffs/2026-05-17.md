## Decisions made and why

1. **Each Codex follow-up addressed in-branch before merge, not as
   separate PRs.** The user asked for the same flow each time
   ("merge" → next PR), so bundling the fix into the same PR
   minimised PR churn AND let the regression tests land alongside
   the fix. Caveat: this only worked because each follow-up was
   bounded and didn't snowball — once a finding required a bigger
   reshape (e.g. the 2nd-brain MCP needing new tools server-side),
   we punted that to a separate Build task.

2. **Wrote the 2nd-brain MCP client to the SPEC §10.1 tool names
   (`archive_search`, `record_pick`), not to what 2nd-brain actually
   exposes.** As of `unsubject/2nd-brain@0a25146`, the MCP only has
   `search_brain` / `get_entry` / `list_recent` / `save_session` —
   different corpus (journal entries, not essays/episodes). Mapping
   the client onto `search_brain` would have given a live integration
   immediately but broken the SPEC contract. Instead, the SPEC §10.2
   graceful fallback (returns `[]` → `archive_overlap=0`) means the
   client degrades cleanly until 2nd-brain ships the proper tools.
   New Build task added on the 2nd-brain side.

3. **Poisson stddev floor for `temperature.ts` z-score.** A quiet
   domain where every other cluster has identical `item_count` gives
   PG `STDDEV()` = 0. Without a floor, a runaway cluster silently
   labels as `warm` because z = 0 (division by 0 guard returns 0).
   Fix: `effectiveStddev = max(observedStddev, sqrt(max(mean, 1)))`
   — the natural Poisson lower bound for count data. Pure helper
   `computeVolumeZ` extracted for direct unit testing.

4. **FP epsilon at ceiling threshold comparisons.** `1.2 / 1.5` is
   `0.7999999999999999` in IEEE 754, so the `≥ 0.8` alert-threshold
   check silently failed at the configured threshold. Added
   `COMPARISON_EPSILON = 1e-9` (six orders of magnitude below the
   minimum representable spend in `numeric(10,6)`) so it cannot mask
   real spend. Regression test pins both `pctOfCeiling < 0.8` and
   `atAlertThreshold === true` so a future "let's drop the epsilon"
   change fails loudly.

5. **`clusterThreshold` → `clusterSimilarityThreshold` + new
   `clusterJoinDistance` for unit clarity.** SPEC §8 speaks in cosine
   similarity (0.70, 0.72, etc.); `src/scoring/cluster.ts` speaks in
   distance (joins when `distance < threshold`). The old helper name
   was ambiguous — passing it straight into `cluster.ts` would have
   over-merged dramatically. Rename forces the unit at the call site
   and adds the corresponding `1 - similarity` helper for the join
   path. No production wiring used the old name yet.

6. **`exclusive.ts` collapses to first-per-source via `DISTINCT ON
   (s.id)` before taking the first two.** Previous `LIMIT 2` returned
   two rows from the same first publisher if that publisher posted
   multiple items before any other outlet did, computing a bogus
   intra-source head start. A genuine exclusive scoop was wrongly
   marked non-exclusive. CTE-based fix; regression test pins the
   multi-item-same-source case explicitly.

7. **Convert `interface ClusterRow / ClusterItemRow` to `type`
   aliases for `db.execute<T>`.** drizzle's generic requires
   `T extends Record<string, unknown>`, and TS treats `interface`
   declarations as closed — fails the constraint. Type aliases of
   object literals are structurally compatible. The rest of the
   codebase inlines literals at the call site; named aliases were
   kept here only because the row shape is referenced from multiple
   places. Saved as `drizzle_execute_type_alias` memory.

  (Gemini=$0.001, Sonnet=$0.008) and stub costs (Gemini=$0.0006,
