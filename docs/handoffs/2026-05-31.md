## Context

Started from "the pipeline keeps tripping the cost ceiling and curate is throwing on a stray trailing comma." Surface bug, but the army-of-agents fragility audit (4 then 7 review agents) surfaced the underlying classes: no fallback when a curate model 5xxs; ceiling sized to median not 95th pct; only the master daily ceiling, no per-stage budget; the orphan-reaper was reaping live runs because it didn't distinguish which process owns which `kind`. Sized into Phase 0 → Phase 3 + an audit-fixes PR + a heartbeat-vs-progress follow-up.

## Decisions made and why

1. **Cost ceiling at $2.20 (μ+1.645σ over the last 30 days), not the median.** A ceiling at the median fires on a normal Tuesday — it stops being a guardrail and becomes a soft daily quota. The 95th percentile fires on anomalies. Also dropped recalibrate prior strength `k` from 20→5 because k=20 was so strong that 30d of actual feedback barely moved sources' authority scores away from seed. Saved to memory as `feedback_ceilings_high_percentile`. (Phase 0, PR #108)

2. **LiteLLM fallback chain on 5xx/rate-limit/timeout/APIError — NOT on BadRequestError.** Per BerriAI/litellm#12503, BadRequestError is treated as a deterministic input failure and LiteLLM deliberately does not fall back on it. The class we care about most (model_name missing from `model_list` after a config update) reads as BadRequestError. Compensating controls: (a) deploy-vps always recycles the litellm container with `--force-recreate --no-deps --wait --wait-timeout 120` so config drift can't persist past a deploy; (b) `scripts/preflight.ts` probes every model in the chain so a missing model is caught BEFORE traffic hits it, not at first user request. (Phase 1, PR #109)

3. **Orchestrator concurrency via `pg_try_advisory_lock` on a postgres-js `raw.reserve()` pinned connection, NOT a DB row lock.** Two cron ticks + an MCP `run_now` racing each other can double-spend the cost ceiling on the same clusters. Advisory locks are session-scoped so the lock has to live on a reserved connection that we hand through the entire run lifetime. Pattern: `withRunLock(raw, async () => runScoring(...))` for cron callers; `runNow()` mirrors the dance manually because it does fire-and-forget with the lock held across the background promise. The reserved-connection-death case is tracked as #113 — open, not in scope this session. (Phase 2.a-b, PR #110)

4. **Boot reaper is now kind-scoped per process, not unconditional.** The reaper used to live in `src/index.ts` and reap ALL `runs.status='running'` on every app boot. Problem: ingestion-worker restart mid-orchestrator-run, app process keeps running fine, runs row gets reaped, then the still-running scoring path keeps inserting candidates against a row marked `failed/process_restart`. Split:
   - `reapOrphanedRunsOnBoot` in ingestion-worker → kind IN ('morning','afternoon')
   - `reapOrphanedManualRunsOnBoot` in app → kind = 'manual'
   - 90-min watchdog cron continues to catch the cross-process orphans (rare).
   Both wrapped in a 30s timeout race so a hung reaper at boot can't gate Fastify/scheduler startup. (Phase 2.a-b PR #110 + Codex follow-up in fix branches)

5. **Per-stage sub-budgets, scope-suffixed in `CostCeilingHitError.code`.** Daily ceiling alone doesn't catch "summarise burned $1.80 and the rest of the stages still have to run." Added a `bucket` column + `assertWithinCeiling(db, projected, bucket)`. `CostCeilingHitError.code` is now `'cost_ceiling_hit:${'daily' | CostBucket}'`, constructed at throw. `runs.error` automatically carries the scope because the orchestrator's catch records `err.code`. Tests updated to assert the scope-suffixed variant. (Phase 3, PR #112)

6. **`generateAllFeeds()` partial-failure tolerant, return order preserved via `Map<slug, path> + orderedSlugs`.** Initial fix made write order = return order, which broke the documented `[all.xml, economy.xml, ...]` contract that callers + the RSS test pin on. Refactored to collect into `writtenByFeed = new Map<string, string>()` keyed by slug, then `orderedSlugs = ['all', ...Object.keys(DOMAIN_CONFIGS)]` rebuilds the documented order independent of write order. (PR #114 + Codex P2 follow-up)

7. **Heartbeat is progress-gated, NOT a standalone timer (#122 / PR #124).** The 30s `setInterval` was tied to event-loop liveness, not work-loop progress — a wedged BullMQ job handler or hung cron callback kept the timer touching the heartbeat file. Replaced with: `lastProgressAt` updated by `markProgress()` calls from real activity; interval skips the touch if `Date.now() - lastProgressAt > progressStaleMs` (default 4× interval = 120s). Wired into BullMQ `worker.on('completed' | 'failed' | 'drained')`, scheduler tick, orchestrator cron (pre-run + post-run), recalibrate, watchdog, scoring tick + compaction. Detection latency = `progressStaleMs + healthcheck maxAge ≈ 4 min`, vs. never previously. Skipped scoring ticks (chainDepth>0 short-circuit from C-P1-1) deliberately do NOT mark progress — a skipped tick IS the wedge signal we want to surface.
