## Decisions made and why

1. **Themes (curated `tags`) are the primary axis; raw keywords secondary.**
   A naive keyword frequency count returns `large-language-models` every day ‚Äî
   arXiv floods ~80% of the rising pool with evergreen ML papers. Tags are
   curated at the curate stage and de-noise the flood for free. (Caveat:
   `ai-safety` tags both real AI-ethics news AND the arXiv flood ‚Äî not a
   perfect separator.)
2. **"Trending" = heat-weighted distinct-CLUSTER count, not row frequency.**
   Deduped by `cluster_id` (+ normalized-headline secondary guard). Weights
   are named constants in `trending.ts` (HEAT: hot 3 / over_sat 2 / warm 1 /
   cold .25; TRAJ: rising 1.5 / peaking 1.25 / new 1 / declining .5) ‚Äî
   deliberately simple for v1, to be tuned against real boards.
3. **Morning-only push, board-only (no LLM titles).** A once-a-day planning
   signal; afternoon overlaps heavily. No per-run LLM cost ‚Äî keeps the daily
   digest flat-cost. Title/SEO shaping stays an on-demand human+Claude step.
4. **MarkdownV2 code spans, not `escapeMarkdownV2`, for kebab terms.**
   Escaping `strait-of-hormuz` ‚Üí `strait\-of\-hormuz`, and inside a code span
   those backslashes render literally. `codeSpan()` carries the raw term
   (stripping backtick/backslash). Regression-tested.
5. **Degrade-not-block.** The orchestrator wraps `computeTrending` in
   try/catch; a trending failure still sends the per-domain summary line.
6. **Honest scope boundary, stated in tool description + module header:**
   editorial descriptors for *topic timeliness*, NOT platform search-volume /
   SEO terms (RSS-only, no platform data).

## Open questions

1. **Weight tuning** ‚Äî watch ~1 week of real morning boards: do the de-noised
   themes stay news-dominant, or does the `ai-safety` arXiv tag creep to the
   top? Adjust HEAT_WEIGHT/TRAJ_WEIGHT in `src/scoring/trending.ts` if so.
   (The two rank-assertion tests are coupled to these constants by design.)
2. **Cross-run candidate duplication (root cause, confirmed this session)** ‚Äî
   the pipeline re-mints a candidate row for a persisting story EVERY run and
   never supersedes the old one (there is no `UPDATE candidates` anywhere), so
   multiple `status='new'` rows per story coexist in-window. The user saw one
   story 4‚Äì5√ó in `search_candidates`. The trending tool dedups by `cluster_id`
   as a workaround, but `list`/`search` still surface the dups. Consider
   superseding/expiring prior candidate rows when a new run re-mints the same
   cluster. Relates to the source-diversity/over-merge item (Build task
   `Y1pITC1kU2piT0daYUJCTg`).
3. **Optional title-suggestion step** ‚Äî if the on-demand shaping proves
   insufficient, scope an LLM title-suggestion tool. Deliberately kept out of
   the daily push to keep cost flat.
